# Feature Specification: 圖書館座位地圖與預測系統

**Feature Branch**: `001-library-seat-tracker`
**Created**: 2025-11-01
**Status**: Draft
**Input**: User description: "圖書館座位地圖與預測系統 — 以互動地圖呈現各圖書館即時座位資訊與開館倒數，提供「附近可用館別推薦」，並預測未來 30 分鐘與 1 小時的可用座位數，協助使用者快速決策與導航"

## Clarifications

### Session 2025-11-01

- Q: 排序按鈕行為與預設狀態 → A: 按鈕為切換式，預設「依距離」排序（近→遠，1000m 內顯示公尺，其餘顯示公里），點擊切換為「依可用座位」（多→少，同分以距離次排序），再點擊切回「依距離」
- Q: 定位權限被拒絕時的處理方式 → A: 顯示提示「請開啟定位以使用距離排序」並附上「去開啟」按鈕，點擊後跳轉至裝置系統設定頁（定位權限設定）；列表自動改以可用座位排序
- Q: 無定位權限時從「依可用座位」切換回「依距離」的行為 → A: 使用者點擊排序按鈕嘗試切換時，系統顯示提示「請開啟定位以使用距離排序」並附上「去開啟」按鈕，點擊後跳轉至系統設定；排序維持「依可用座位」不變
- Q: 閉館狀態的視覺呈現 → A: 圖書館已閉館（超過閉館時間或尚未開館）時，詳細視窗以灰底呈現「今日已閉館」或「尚未開館」狀態
- Q: 節假日或臨時公告的資料來源 → A: MVP 不整合節假日/臨時公告 API，所有營業時段註記「特殊時段請以官網為準」；Phase 2 若有外部 API 再整合
- Q: 預測模型的訓練與更新機制 → A: 每 10 分鐘執行資料擷取，同時更新 realtime table 與插入 history table；每天執行預測排程重新訓練模型，若預測結果更好則採用新模型；模型使用最近 30 天歷史資料訓練
- Q: 距離顯示單位規則 → A: 1000 公尺內以公尺為單位顯示，1000 公尺以上以公里為單位顯示

## User Scenarios & Testing

### User Story 1 - 快速查找附近有空位的圖書館 (Priority: P1)

學生 Amy 準備去圖書館讀書，想快速找到附近且現在有位子的館別，以便立即前往。

**Why this priority**: 這是系統的核心價值主張——協助使用者立即做出決策。沒有這個功能，系統無法提供基本價值。

**Independent Test**: 可以完整測試：使用者打開應用程式，查看地圖上各館別的座位狀態標記，選擇最近的有空位館別並取得導航資訊。這個流程本身就能提供完整的使用者價值。

**Acceptance Scenarios**:

1. **Given** 使用者允許定位權限且附近有多個圖書館，**When** 使用者開啟應用程式，**Then** 地圖顯示所有圖書館標記，標記顏色反映即時座位狀態（綠色 = 空位充足、黃色 = 空位少、灰色 = 已滿、白框灰圈 = 無資料），且顯示使用者當前位置標記（藍色圓點與半透明光圈）
2. **Given** 使用者允許定位權限，**When** 使用者開啟應用程式，**Then** 地圖介面顯示「已定位」狀態指示器
3. **Given** 使用者拒絕定位權限或定位失敗，**When** 使用者開啟應用程式，**Then** 地圖不顯示使用者位置標記，且顯示定位錯誤提示訊息
4. **Given** 使用者在地圖上看到標記，**When** 使用者點擊綠色標記，**Then** 底部資訊框顯示館名與「查看詳細資訊」按鈕
5. **Given** 使用者點擊了標記並看到資訊框，**When** 使用者點擊「查看詳細資訊」，**Then** 開啟詳細視窗，顯示館名、地址（含 Google 導航連結）、電話、營業時間、與使用者距離、現有/總座位數
6. **Given** 使用者在詳細視窗中，**When** 使用者點擊地址的導航連結，**Then** 開啟 Google 地圖導航
7. **Given** 使用者在地圖畫面，**When** 使用者點擊地圖空白處或關閉按鈕，**Then** 底部資訊框回到僅顯示圖例的狀態

---

### User Story 2 - 透過列表排序快速比較館別 (Priority: P1)

使用者 Bob 想比較多個圖書館的座位資訊，並根據距離或可用座位數排序，以做出最佳選擇。

**Why this priority**: 列表排序是核心比較功能，與地圖視覺化互補，共同構成 MVP。使用者可以透過列表快速掃描所有選項而不用逐一點擊地圖標記。

**Independent Test**: 可以獨立測試：使用者開啟右上角列表視窗，切換「依距離」或「依可用座位」排序，點擊列項查看詳細資訊。測試完成後可驗證排序邏輯是否正確且資訊準確。

**Acceptance Scenarios**:

1. **Given** 使用者在主畫面，**When** 使用者展開右上角列表視窗（點擊 ＋），**Then** 列表顯示所有圖書館，每列顯示館名與「現有/總座位」，預設以「依距離」排序（距離近到遠，1000 公尺內顯示公尺，其餘顯示公里）
2. **Given** 使用者允許定位權限且列表以「依距離」排序，**When** 使用者點擊排序按鈕，**Then** 按鈕切換為「依可用座位」，列表改以可用座位總和由多到少排序（同分時以距離次排序）
3. **Given** 使用者允許定位權限且列表以「依可用座位」排序，**When** 使用者點擊排序按鈕，**Then** 按鈕切換回「依距離」，列表改以距離由近到遠排序
4. **Given** 使用者在列表中，**When** 使用者點擊列項，**Then** 開啟該館別的詳細視窗
5. **Given** 使用者未允許定位權限且列表預設顯示時，**When** 系統偵測到無定位權限，**Then** 系統顯示提示「請開啟定位以使用距離排序」並附上「去開啟」按鈕；列表自動以「依可用座位」排序
6. **Given** 使用者未允許定位權限且列表以「依可用座位」排序，**When** 使用者點擊排序按鈕嘗試切換為「依距離」，**Then** 系統顯示提示「請開啟定位以使用距離排序」並附上「去開啟」按鈕，點擊後跳轉至系統設定；排序維持「依可用座位」不變

### User Story 3 - 查看開館倒數以規劃時間 (Priority: P2)

上班族考生 Carol 想知道下班後哪些圖書館仍開放且有空位，以便規劃讀書時間。

**Why this priority**: 開館倒數對時間敏感的使用者提供重要價值，但非所有使用者的首要需求。P1 功能已能提供基本服務，此功能強化時間規劃體驗。

**Independent Test**: 可以獨立測試：使用者查看詳細視窗中的營業時間與倒數提示，驗證剩餘時間計算正確，且在 ≤60 分鐘與 ≤15 分鐘時顯示明顯提示。

**Acceptance Scenarios**:

1. **Given** 使用者在詳細視窗中，**When** 圖書館距離閉館時間 > 60 分鐘，**Then** 顯示正常營業時間資訊
2. **Given** 使用者在詳細視窗中，**When** 圖書館距離閉館時間 ≤ 60 分鐘且 > 15 分鐘，**Then** 顯示「即將閉館」提示
3. **Given** 使用者在詳細視窗中，**When** 圖書館距離閉館時間 ≤ 15 分鐘，**Then** 以紅色或明顯警示樣式顯示倒數時間
4. **Given** 使用者在詳細視窗中，**When** 圖書館已閉館（當前時間超過閉館時間或尚未開館），**Then** 以灰底呈現「今日已閉館」或「尚未開館」狀態
5. **Given** 使用者查看營業時間，**When** 顯示營業時段，**Then** 註記「特殊時段請以官網為準」（MVP 不整合節假日或臨時公告 API）

---

### User Story 4 - 查看座位數預測以提前規劃 (Priority: P2)

使用者 David 在前往圖書館前 30-60 分鐘，想知道預計抵達時是否還有空位，以決定是否出門。

**Why this priority**: 預測功能為進階需求，增強使用者信心但非必要。P1 功能已提供即時資訊，預測功能提升體驗但需要額外資料收集與模型訓練。

**Independent Test**: 可以獨立測試：系統定期擷取座位資料並訓練預測模型，前端呼叫 /predict API 取得未來 30 分鐘與 1 小時的預測座位數，驗證預測結果是否合理。

**Acceptance Scenarios**:

1. **Given** 系統已收集足夠歷史資料，**When** 使用者查詢某館別預測，**Then** 系統回傳未來 30 分鐘與 60 分鐘的預測可用座位數
2. **Given** 預測模型運作正常，**When** 使用者查看預測結果，**Then** 預測值顯示於詳細視窗中（若適用）
3. **Given** 預測模型不可用或資料不足，**When** 使用者查詢預測，**Then** 系統以平均數呈現
4. **Given** 使用者停留頁面超過 15 分鐘，**When** 系統偵測到資料可能過期，**Then** 提示「資料可能已過期，是否立即更新」

---

### User Story 5 - 自動更新資料以確保資訊即時性 (Priority: P3)

系統需定期更新座位資料，確保使用者看到的資訊不超過 10 分鐘。

**Why this priority**: 資料更新機制是背景功能，支撐所有前述功能的資料時效性。雖然重要但對使用者較不可見，屬於技術支撐層。

**Independent Test**: 可以獨立測試：驗證伺服器端每 10 分鐘擷取外部 API，前端每 10 分鐘重新拉取 /realtime 與 /predict，資料時間戳正確更新。

**Acceptance Scenarios**:

1. **Given** 系統運作中，**When** 每 10 分鐘觸發擷取排程，**Then** 系統呼叫外部座位 API 並將同一輪擷取的資料標記為同一批次以便追蹤
2. **Given** 外部 API 呼叫失敗，**When** 系統偵測到失敗，**Then** 間隔 1 分鐘重試，最多 3 次
3. **Given** 前端頁面開啟，**When** 每 10 分鐘輪詢時間到，**Then** 前端重新拉取 /realtime 與 /predict
4. **Given** 前端頁面非使用中（hidden），**When** 輪詢時間到，**Then** 暫停輪詢以節省資源
5. **Given** 使用者希望立即更新，**When** 使用者點擊「重新整理」按鈕，**Then** 立即拉取最新資料

---

### Edge Cases

- **定位權限被拒絕（初次開啟應用程式）**：地圖不顯示使用者位置標記，顯示定位錯誤提示訊息，使用預設位置（台北市政府）作為地圖中心點
- **定位權限被拒絕（初次開啟列表）**：系統顯示提示「請開啟定位以使用距離排序」並附上「去開啟」按鈕，點擊後跳轉至系統設定頁；列表自動以可用座位排序
- **定位權限被拒絕（嘗試切換排序）**：使用者從「依可用座位」嘗試切換回「依距離」時，系統顯示提示「請開啟定位以使用距離排序」並附上「去開啟」按鈕，點擊後跳轉至系統設定；排序維持「依可用座位」不變
- **定位服務不可用或超時**：視為定位失敗，使用預設位置（台北市政府），顯示錯誤提示並提供重試按鈕
- **單館資料缺漏**：該館標記為白色外框灰圈（未提供座位資料），列表顯示「—」
- **外部 API 無回應或超時**：維持上次快照並顯示「資料延遲」徽章
- **預測模型失敗或資料不足**：系統以統計方法（移動平均）呈現預測值
- **異常座位數值**：free_count > total_count 或 < 0 視為異常剔除
- **節假日或臨時公告**：MVP 不整合外部節假日 API，所有營業時段註記「特殊時段請以官網為準」
- **圖書館已閉館**：詳細視窗以灰底呈現「今日已閉館」或「尚未開館」狀態

## Requirements

### Functional Requirements

#### 地圖與標記

- **FR-001**: 系統必須在地圖上顯示所有圖書館標記，每個標記代表一個圖書館
- **FR-002**: 系統必須根據可用座位數為標記賦予顏色：綠色（freeCount > 10）、黃色（0 < freeCount ≤ 10）、灰色（freeCount = 0）、白色外框灰圈（無座位資料）
- **FR-003**: 使用者必須能夠點擊地圖標記以查看該館的基本資訊
- **FR-004**: 系統必須在底部資訊框常駐顯示圖例，說明各顏色代表的座位狀態
- **FR-004-1**: 系統必須在地圖上顯示使用者當前位置標記（藍色圓點），標記尺寸 16x16px，外圍半透明光圈 radius 32px
- **FR-004-2**: 定位成功時，使用者位置標記必須顯示在地圖上；定位失敗或未授權時，不顯示使用者位置標記
- **FR-004-3**: 系統必須在地圖介面提供定位狀態視覺回饋（成功時顯示「已定位」指示器，失敗時顯示錯誤提示）

#### 底部資訊框

- **FR-005**: 未選點時，底部資訊框僅顯示圖例
- **FR-006**: 選點後，底部資訊框顯示館名與「查看詳細資訊」按鈕
- **FR-007**: 使用者必須能夠透過點擊地圖空白處或關閉按鈕回到未選點狀態

#### 右上角列表視窗

- **FR-008**: 系統必須提供右上角可收合的列表視窗，預設疊在地圖右上角
- **FR-009**: 列表視窗必須支援收合（－）與展開（＋）控制
- **FR-010**: 展開時，列表顯示所有圖書館，每列顯示館名、與使用者距離、「現有/總座位」
- **FR-011**: 排序按鈕為切換式按鈕，預設顯示「依距離」，點擊後切換為「依可用座位」，再點擊切換回「依距離」（若有定位權限）
- **FR-012**: 依距離排序時，系統計算使用者與館別之間的實際地理距離，1000 公尺內以公尺為單位，1000 公尺以上以公里為單位（由近到遠排序）
- **FR-013**: 若使用者未允許定位權限，系統顯示提示「請開啟定位以使用距離排序」並附上「去開啟」按鈕，點擊後跳轉至裝置系統設定頁（定位權限設定）；列表自動改以可用座位排序
- **FR-013-1**: 若使用者未允許定位權限且嘗試從「依可用座位」切換回「依距離」排序，系統顯示提示「請開啟定位以使用距離排序」並附上「去開啟」按鈕；排序維持「依可用座位」不變，直到使用者授予定位權限
- **FR-014**: 依可用座位排序時，列表依可用座位總和由多到少排序，同分時以距離作次排序
- **FR-015**: 使用者必須能夠點擊列項以開啟該館的詳細視窗

#### 詳細資訊視窗

- **FR-016**: 詳細視窗必須顯示館名、地址、電話、營業時間、與使用者距離、現有/總座位數
- **FR-017**: 地址必須包含可點擊的 Google 導航連結
- **FR-018**: 電話必須支援點擊撥號（若裝置支援）
- **FR-019**: 使用者必須能夠透過頂部 X 按鈕關閉詳細視窗回到地圖

#### 開館狀態與倒數

- **FR-020**: 系統必須根據當前時間與館別營業時段計算剩餘開館時間
- **FR-021**: 剩餘時間 ≤ 60 分鐘時，系統顯示「即將閉館」提示
- **FR-022**: 剩餘時間 ≤ 15 分鐘時，系統以紅色或明顯警示樣式顯示倒數時間
- **FR-023**: 圖書館已閉館時（當前時間超過閉館時間或尚未開館），詳細視窗以灰底呈現「今日已閉館」或「尚未開館」狀態
- **FR-024**: 營業時間顯示必須註記「特殊時段請以官網為準」（MVP 不整合節假日或臨時公告 API）

#### 資料更新與輪詢

- **FR-025**: 伺服器端必須每 10 分鐘擷取外部座位 API
- **FR-026**: 擷取失敗時，系統間隔 1 分鐘重試，最多 3 次
- **FR-027**: 同一輪擷取的資料必須標記為同一批次以便追蹤與回溯
- **FR-028**: 擷取到的座位資料必須同時更新即時表（realtime table）與插入歷史表（history table）並記錄時間戳
- **FR-029**: 前端必須每 10 分鐘重新拉取 /realtime 與 /predict API
- **FR-030**: 前端頁面非使用中（hidden）時，暫停輪詢
- **FR-031**: 系統必須提供「重新整理」按鈕供使用者立即更新
- **FR-032**: 使用者停留頁面超過 15 分鐘時，系統提示「資料可能已過期，是否立即更新」

#### 座位預測

- **FR-033**: 系統必須預測未來 30 分鐘與 1 小時的可用座位數（以館別聚合）
- **FR-034**: 預測輸入為過去 2 小時的座位快照（12 筆，每 10 分鐘一筆）
- **FR-035**: 系統每天執行預測排程，重新訓練模型，若預測結果更好則採用新模型
- **FR-036**: 預測模型不可用或資料不足時，系統以統計方法（移動平均）回退呈現預測值
- **FR-037**: 預測模型使用最近 30 天歷史資料訓練

#### 資料清理與異常處理

- **FR-038**: 系統必須過濾異常值：free_count > total_count 或 < 0 視為異常剔除
- **FR-039**: 資料缺漏時，系統以同區域前次有效值補值
- **FR-040**: 外部 API 無回應或超時時，維持上次快照並顯示「資料延遲」徽章
- **FR-041**: 單館資料缺漏時，該館標記為白色外框灰圈，列表顯示「—」

#### API 合約

- **FR-042**: 系統必須提供 GET /realtime API，回傳每館聚合的可用座位總數/總座位數與資料時間戳
- **FR-043**: 系統必須提供 GET /predict API，回傳 30 分鐘與 60 分鐘的預測可用座位數
- **FR-044**: 系統必須提供 GET /libraries API，回傳館別基本資料（名稱、地址、電話、經緯度、營業時段）
- **FR-045**: 系統必須提供 GET /suggestions API，依 user_lat、user_lng、sort_by 參數回傳排序後的館別清單

#### 無障礙與易用性

- **FR-046**: 地圖標記與按鈕必須具備可辨識的 ARIA 標籤
- **FR-047**: 標記顏色設計必須色盲友善（綠/灰/白外框組合），同時以圖例與文字輔助
- **FR-048**: 主要互動元件必須支援 Tab 鍵順序，Enter 鍵觸發選取

### Key Entities

- **Library（圖書館）**: 代表一個圖書館分館，包含館名、地址、電話、經緯度、營業時段等基本資訊
- **RealtimeSeatStatus（即時座位狀態）**: 儲存各館最新的座位快照，供前端查詢即時資訊，包含館名、區域 ID、樓層、區域名稱、可用座位數、總座位數、更新時間戳
- **HistorySeatStatus（歷史座位狀態）**: 儲存所有座位快照的歷史記錄，供預測模型訓練使用，包含館名、區域 ID、樓層、區域名稱、可用座位數、總座位數、記錄時間戳
- **PredictionResult（預測結果）**: 代表對某館在特定時間範圍（30 分鐘或 60 分鐘）的預測可用座位數，包含目標時段、預測值、生成時間、有效時間

## Success Criteria

### Measurable Outcomes

- **SC-001**: 使用者能在 2 秒內載入地圖與館別標記（快取命中時）
- **SC-002**: 系統能在 500 毫秒內繪製完所有館別標記與使用者位置標記
- **SC-003**: 使用者能在 3 次點擊內從地圖到取得導航資訊（點擊標記 → 查看詳細 → 點擊導航）
- **SC-004**: 座位資料更新頻率維持在 10 分鐘內，95% 的擷取請求成功
- **SC-005**: 預測準確度維持在可接受範圍內（具體數值需依實際資料評估）
- **SC-006**: 系統可用性達到 99.5% SLA
- **SC-007**: 使用者定位拒絕情況下，仍能正常使用排序與查詢功能（改以座位或行政區排序）
- **SC-008**: 外部 API 失敗時，系統能維持上次快照並提示使用者，不影響核心查詢功能
- **SC-009**: 定位成功時，使用者能在地圖上清楚看到自己的位置標記與定位狀態指示器
- **SC-010**: 定位失敗時，使用者能在 1 秒內看到明確的錯誤提示與重試選項

## Assumptions

- 外部座位 API 每 10 分鐘更新一次為合理頻率，符合圖書館座位變化速度
- 使用者裝置支援 GPS 定位，若拒絕權限則提供跳轉至系統設定的功能，並改以可用座位排序
- Google 地圖導航連結為標準導航方式（可依實際需求調整為其他地圖服務）
- 預測排程每天執行一次，重新訓練模型並評估是否採用新模型（依預測準確度決定）
- 預測模型回退策略使用統計方法（移動平均），當模型不可用或資料不足時啟用
- MVP 不整合節假日或臨時公告 API，所有營業時段註記「特殊時段請以官網為準」；Phase 2 若有外部 API 再整合
- 系統不保存使用者個資，定位座標僅用於即時排序計算，不落資料庫
- 座位資料同時維護兩張表：realtime table（最新快照，供前端查詢）與 history table（歷史記錄，供模型訓練）

## Out of Scope (Phase 2)

- 區域層級視覺化（同館不同樓層/區域）
- 尖峰時段熱力時間軸
- 推播提醒（如「常去的館別現在有空位」）
- 館別臨時公告 API 整合（若未來取得官網 API）
